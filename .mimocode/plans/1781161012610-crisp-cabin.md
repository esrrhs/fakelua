# FakeLua 可改进点分析

## 1. 头文件 includes：common.h 过度包含

**文件**: `src/util/common.h`

`common.h` 作为"万能头文件"包含了约 60 个系统头文件，其中大部分未被使用。例如 `<condition_variable>`, `<mutex>`, `<shared_mutex>`, `<thread>`, `<future>`, `<regex>`, `<random>`, `<queue>`, `<stack>`, `<deque>`, `<list>`, `<set>`, `<signal.h>`, `<dirent.h>`, `<execinfo.h>` 等。这会导致：
- 编译时间显著增加（每个 `.cpp` 都解析这些头文件）
- 不必要的宏污染和符号冲突风险
- 违反最小包含原则

**建议**: 将 `common.h` 改为只包含各模块实际需要的头文件，每个 `.cpp`/`.h` 按需包含。

---

## 2. NativeToFakelua/FakeluaToNative 模板分发链冗长

**文件**: `include/fakelua.h:346-429`, `include/fakelua.h:449-489`

两个模板函数各有 16 个 `if constexpr` 分支，每个分支对应一个具体类型。代码量大且重复度高。

**建议**: 使用类型 trait 映射表或折叠表达式简化。例如定义一个 `TypeTag` trait：
```cpp
template<typename T> struct NativeTypeTag;
template<> struct NativeTypeTag<int> { static constexpr int value = 0; };
// ... 声明所有类型
template<typename T>
CVar NativeToFakelua(State* s, T v) {
    return NativeToFakeluaDispatch<T>(s, v, std::make_index_sequence<type_count>{});
}
```

---

## 3. Call() 模板中硬编码 8 个 reinterpret_cast 分支

**文件**: `include/fakelua.h:539-559`

`Call()` 函数对 0-8 个参数各写了一个 `reinterpret_cast<CVar (*)(CVar, CVar, ...)>(addr)(...)` 分支，逻辑完全相同只是参数个数不同。

**建议**: 用参数包展开 + constexpr if 构建函数指针类型：
```cpp
using FuncType = CVar (*)(/* expanded CVar types via pack */);
ret_var = reinterpret_cast<FuncType>(addr)(inter::NativeToFakelua(s, std::forward<Args>(args))...);
```
但 C++ 不支持"参数包长度的函数指针类型"直接构造，可以用模板递归或 std::index_sequence 辅助。实际最简洁的方案是用固定数组 + 转发。

---

## 4. CGen 生成的 C 代码中 quick_data 线性扫描展开为 8 个 if

**文件**: `src/compile/c_gen.cpp:300-308`, `408-415`, `447-454`

`FlGetTable`、`FlSetTable`（删除路径）、`FlSetTable`（更新路径）中，quick_data 的线性扫描被手动展开为 8 个 `if` 语句。三处重复了相同的展开模式。

**建议**: 用 C 的 `for` 循环替代手动展开，编译器 `-O3` 会自动展开小循环。或者用宏统一定义：
```c
#define QUICK_DATA_SCAN(tbl, h, body) \
    for (uint32_t _i = 0; _i < (tbl)->count_ && _i < 8; ++_i) { \
        if ((tbl)->quick_data_[_i].hash == (h)) { body } \
    }
```

---

## 5. common.h 包含 `<assert.h>` 和 `<cassert>` 重复

**文件**: `src/util/common.h:4-5`

同时包含了 C 风格 `<assert.h>` 和 C++ 风格 `<cassert>`，且项目使用的是自定义 `DEBUG_ASSERT` 宏（`src/util/debug.h`）。C 标准 `assert()` 在 Release 模式被 `NDEBUG` 消除，而 `DEBUG_ASSERT` 也做了同样的事。

**建议**: 移除 `<assert.h>` 和 `<cassert>`，统一使用 `DEBUG_ASSERT`。

---

## 6. LOG_INFO/LOG_ERROR 宏缺少花括号保护

**文件**: `src/util/logging.h:22-29`

```cpp
#define LOG_INFO(fmt, ...) \
    if (fakelua::CheckLogLevel(fakelua::LogLevel::Info)) { \
        fakelua::Log(...); \
    }
```

`if` 语句没有 `do { } while(0)` 包裹，在 `if-else` 语义中会导致悬空 else 问题：
```cpp
if (condition)
    LOG_INFO("msg");
else
    something();
// else 会匹配 LOG_INFO 内部的 if，而非外部的 if
```

**建议**: 改为 `do { if (...) { ... } } while(0)` 模式。

---

## 7. common.h 包含平台特定头文件未做条件编译

**文件**: `src/util/common.h:14,28,50,57`

- `<dirent.h>` (POSIX only)
- `<pthread.h>` (POSIX only)
- `<unistd.h>` (POSIX only)
- `<execinfo.h>` (已用 `#ifndef _WIN32` 保护)

但 `<dirent.h>`、`<pthread.h>`、`<unistd.h>` 未做 Windows 保护。虽然 CMakeLists.txt 中 Windows 编译使用 MinGW（有这些头文件的兼容实现），但这不规范。

**建议**: 对 POSIX 特有头文件统一加 `#ifndef _WIN32` 保护。

---

## 8. VarTable 缺少拷贝/移动语义

**文件**: `src/var/var_table.h:35-36`

`VarTable` 禁止了拷贝，但没有显式声明移动构造/赋值。在某些场景（如函数返回 table）会退化为拷贝整个 table 或使用 shared_ptr 包装。

**建议**: 显式声明移动操作，利用 move-only 语义避免不必要的 deep copy。

---

## 9. VarString 使用零长数组（flexible array member）

**文件**: `src/var/var_string.h:46`

```cpp
char data_[0];
```

零长数组是 GCC 扩展，不是标准 C++。虽然 MSVC 也有类似支持，但严格来说不可移植。

**建议**: 改用 C++20 的 `std::array` + placement new，或者使用 `char data_[1]` 配合 `sizeof(VarString)` 手动计算偏移（更兼容）。

---

## 10. GccJitter::Compile 中 fork/exec 未处理 SIGCHLD

**文件**: `src/jit/gcc_jit.cpp:184-209`

`fork()` + `waitpid()` 是正确的，但如果进程在 `fork()` 之后、`waitpid()` 之前收到 SIGCHLD 信号，默认行为可能干扰。在信号处理不当的环境下可能产生僵尸进程。

**建议**: 在 fork 前临时屏蔽 SIGCHLD，或使用 `posix_spawn`（更安全、更跨平台）。

---

## 11. GccJitter 跨平台进程创建可抽象

**文件**: `src/jit/gcc_jit.cpp:84-228`

`GccJitter::Compile` 中有大量平台分派代码：Linux 用 fork+execvp+dlopen，Windows 用 _spawnvp+LoadLibraryA，macOS 有额外的 `-undefined dynamic_lookup` 选项。

**建议**: 抽象出一个 `ProcessRunner` 工具类，封装"写临时文件→编译→加载动态库→查找符号"的流程，减少平台分派代码的重复。

---

## 12. 缺少线程安全文档和工具

**文件**: `src/state/state.h:14-19`, `src/jit/vm.h:10-14`

代码注释明确说明 State/Vm 是单线程实体，但没有提供线程安全的使用指南或辅助工具。用户可能在多线程环境中错误共享 State。

**建议**: 
- 在 `fakelua.h` 的 `FakeluaNewState` 文档中明确标注"非线程安全"
- 考虑提供一个 `FakeluaThreadLocalState` 包装器，自动管理每线程的 State 实例

---

## 13. ConstString 使用 `string_view` 作为 map key 存在悬挂引用风险

**文件**: `src/state/const_string.h:31`

```cpp
std::unordered_map<std::string_view, int64_t> str_to_id_map_;
```

`string_view` 不拥有数据，如果原始字符串被释放，map 中的 key 将变成悬挂引用。虽然当前实现中字符串来自 HeapAllocator（生命周期长于 ConstString），但这种隐式依赖很脆弱。

**建议**: 改为 `std::unordered_map<std::string, int64_t>` 或确保文档中说明 string_view 的生命周期约束。

---

## 14. 测试框架缺少对异常路径的覆盖

**文件**: `test/`

从 test 目录结构看，大部分测试覆盖的是正常编译路径。缺少对以下异常场景的测试：
- 超过 8 个参数的函数调用
- 不支持的语法（goto、label、varargs）的错误信息验证
- 编译失败时的错误恢复
- 并发使用 State 的检测

**建议**: 增加异常路径的单元测试，验证错误消息的准确性和异常的安全清理。

---

## 15. 缺少 API 稳定性保证

**文件**: `include/fakelua.h`

公共 API 直接暴露了 `State*` 指针和 `CVar` 结构体，没有 ABI 版本控制。一旦内部结构变化（如 `CVar` 布局调整），所有使用方都需要重新编译。

**建议**: 考虑引入 `fakelua_version()` 函数和 `FAKELUA_ABI_VERSION` 宏，用于运行时/编译时 ABI 兼容性检查。

---

## 16. Benchmark 缺少 warmup 阶段

**文件**: `benchmark/benchmark_algo.cpp`

Google Benchmark 默认会做 warmup，但如果 JIT 编译在第一次调用时触发（TCC/GCC 编译开销），首次调用会显著慢于后续调用，影响测量准确性。

**建议**: 在 benchmark 中显式调用一次函数进行 warmup，确保 JIT 编译在计时开始前完成。

---

## 17. C 代码生成中的宏定义可提取为独立头文件

**文件**: `src/compile/c_gen.cpp:74-500+`

`GenerateHeader()` 中内联了大量 C 宏定义（`SET_NIL`, `SET_BOOL`, `SET_INT`, `VarEqual`, `VarHash`, `FlGetTable`, `FlSetTable` 等），总计约 500 行。这些宏定义与 CGen 的 C++ 代码生成逻辑混在一起。

**建议**: 将这些 C 运行时宏/函数提取到一个独立的 `runtime_header.h` 字符串常量中，CGen 通过 `#include` 引用。这样便于独立维护和测试 C 运行时代码。

---

## 18. 缺少编译时间统计

**文件**: `src/compile/compiler.cpp`

编译管线有多个阶段（解析→预处理→推导→C生成→JIT），但没有记录每个阶段的耗时。对于性能调优和用户反馈很重要。

**建议**: 在 `Compiler::Compile` 中添加 `LOG_INFO` 记录每个阶段的耗时（使用 `std::chrono`）。

---

## 19. VarTable 的 bucket_count_ 为 0 时的快速路径未优化删除

**文件**: `src/compile/c_gen.cpp:406-415`

quick_data 删除时用"与末尾交换+count--"策略，这是 O(1) 的，但会破坏元素的插入顺序。如果用户依赖 table 的遍历顺序（如 ipairs），这可能导致不一致。

**建议**: 在文档中明确说明遍历顺序不保证，或在 quick_data 模式下保持顺序（用 memmove 代替 swap）。

---

## 20. 缺少对 Lua 5.4 兼容性测试套件的集成

**文件**: `test/lua/infer/`

当前测试是手写的 Lua 片段，覆盖了 FakeLua 支持的子集。但没有系统性地从 Lua 5.4 官方测试套件中筛选可兼容的测试用例。

**建议**: 从 Lua 5.4 test suite 中筛选 FakeLua 支持的子集（纯数值计算、简单控制流），建立兼容性回归测试集。
